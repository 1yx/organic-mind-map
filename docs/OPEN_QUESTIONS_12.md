# 开放性问题 (PNG Export 设计)

本文件记录对 `@openspec/changes/png-export/**` 提案与设计进行审查后发现的重大工程陷阱（尤其是前端 Canvas 安全与渲染机制的隐患）。

---

## 1. SVG 转 Canvas 的致命安全限制：外链图片将集体消失 (SVG-as-Image Security Limitation) [已决策]

### 发现的问题
在 `design.md` 的实现方案中，规划的导出管线是：“`XMLSerializer` -> `Blob URL` -> `Image` -> `Canvas`”。
但是，在前面的 `ai-svg-center-visual` (问题 9) 中，我们刚刚决定了：CLI 仅透传中心图的 URL 链接（例如 Iconify 的链接），由前端浏览器将其渲染到导图中心。

### 苏格拉底式问题
* **极其严重的浏览器安全机制陷阱**：当浏览器把一段 SVG 字符串转化为 `Blob URL` 并塞进 `new Image()` 时，浏览器会触发最严格的安全隔离上下文（隔离渲染）。**在这个隔离上下文中，SVG 内部包含的任何外部网络请求（无论是跨域图片 `<image href="http...">` 还是外部 CSS/字体）都会被浏览器无情且静默地拦截并丢弃！**
* 这意味着：虽然用户在网页上看着中心图（外链加载的）好好的，但一旦点击“Export PNG”，下载下来的图片中心将是**完全空白**的！
* **如何破局？** 既然导图必须要导出完美的 PNG，这是否意味着前端在执行 `XMLSerializer` 序列化之前，必须先写一个“资产内联化（Asset Inlining）”的预处理脚本？（即：前端发起 `fetch` 请求把那个外链 SVG 源码拉下来，直接以内联 `<svg>` 或 Base64 DataURL 的形式动态替换掉原来的外链 `<image>` 标签，然后再将整棵 DOM 树交给 Canvas 去画？）

### 决策记录
* **决策结果**: 必须在触发“Export PNG”点击动作、进入 `XMLSerializer` 序列化之前，增加一道强硬的前端“资产内联化（Asset Inlining）”预处理拦截器。
* **决策理由**: 浏览器对于 `<img src="data:image/svg+xml...">` 内部网络请求的封杀是无法通过配置绕过的（这是 W3C 的安全硬性规范）。既然问题 9 决定了由浏览器去负责中心图网络请求，为了保证导出的图片不“开天窗”，前端必须在用户点击导出的那一瞬间，通过 `fetch` 将页面上活跃的外链 SVG 资源下载下来，将其转码为 `Base64 DataURL`（形如 `data:image/svg+xml;base64,...`），并用这个绝对隔离安全的文本替换掉克隆的 DOM 树中原有的外链属性。只有这棵彻底被“内联化（Inlined）”的纯净 SVG DOM 树，才能被 Canvas 100% 完美无损地画出来。
* **后续修改文档**: 在 `png-export` 的任务规范中，强制追加一条导出前置生命周期任务：解析 SVG DOM，提取所有外链资源（如 `image`），执行异步拉取并替换为 DataURL，然后再执行后续的 `XMLSerializer` -> `Canvas` 管线。

## 2. 外部 Web 字体导致的文字错位与乱码 (Web Fonts & Text Rendering Missing) [已决策]

### 发现的问题
与外链图片同理，博赞有机导图上的分支概念文字（Text-on-Path）是排版的核心。

### 苏格拉底式问题
* 如果我们的 `@omm/web` 前端为了好看，加载了外部的 Web 字体（例如 Google Fonts），在网页上由于浏览器提前下载了字体，排版是完美的。
* 但是！一旦执行上面那条 SVG -> Canvas 的序列化管线，外部字体同样会被安全机制屏蔽。Canvas 画出来的字会全部退化成操作系统默认字体（比如宋体/Times New Roman），这不仅极其丑陋，还会导致文字长度突变，瞬间与有机曲线发生严重的穿模和错位！
* **架构底线约束**：在 MVP 阶段，我们是否应该定下一条铁律：**整个 SVG 渲染引擎必须绝对依赖操作系统的内置安全字体（System Fonts，如 `sans-serif` 或 `system-ui`）**，严禁在 SVG 内部使用任何需要网络加载的 Web 字体，以彻底避开“字体 Base64 内联化”这个深水区？

### 决策记录
* **决策结果**: 坚决采纳此防线建议：在 MVP 及未来长期的基础排版引擎中，**绝对禁用任何外部 Web 字体**。整个渲染树的 `font-family` 必须退回到最安全的操作系统内置字体组合（如 `system-ui, -apple-system, sans-serif` 等）。
* **决策理由**: “字体 Base64 内联化”是一个巨大的工程沼泽（Web 字体动辄几 MB 大小，如果每次导出都要在内存里完成切片转码，整个浏览器的内存会直接爆掉）。如果我们允许开发者随意使用 Web 字体，那这套建立在“高频且严苛的 `Canvas 2D measureText`”之上的防碰撞引擎就会形同虚设。当导出为 Canvas 时，一旦因为安全机制导致字体丢失退化为系统字体，文字的宽度会瞬间变化，原先安全的包围盒会全部失效，导致导出的图纸严重穿模。因此，彻底封杀外部字体是最聪明、最具有工程手腕的“做减法”决策，这能在零成本的前提下，保证页面预览与最终 PNG 导出**像素级一致 (Pixel-perfect Match)**。
* **后续修改文档**: 在渲染器（`readonly-svg-renderer`）及导出（`png-export`）模块中追加一条严厉的开发军规：禁止在 SVG、CSS 和 Canvas 测量系统中引入和加载任何形式的远程或本地 `Web Font (@font-face)`。

## 3. 导出分辨率与视网膜屏幕的模糊问题 (Export Resolution & Device Pixel Ratio) [已决策]

### 发现的问题
`design.md` 提到：“export at 2x or 3x preview scale if practical... keep memory usage safe”。

### 苏格拉底式问题
* 我们的逻辑坐标系（A3/A4 ViewBox）非常大，比如 A4 可能是 `2970 x 2100`（相当于 300 DPI 下的像素）。如果直接用这个尺寸创建 Canvas 并绘制，确实能得到一张高清图。
* 但是，我们在用前端代码把 SVG 塞进 `new Image()` 时，如果不对 `Image.width` 和 `Canvas.width` 的缩放比例 (Scale Factor) 进行严密的数学控制，导出的 PNG 会不会在 Retina (MacBook) 高分屏上显得异常模糊？
* 我们是否应该在规范中明确：**Canvas 的物理尺寸（width/height 属性）必须严格等于 ViewBox 的尺寸**，而不是取屏幕上 DOM 元素的实际渲染尺寸，从而确保无论用户在多小的屏幕上预览，导出的 PNG 永远是原生 A3/A4 级别的绝对高清工业图纸？

### 决策记录
* **决策结果**: 维持自适应设计的核心，**只限制导出 Canvas 的纵横比（Aspect Ratio）必须与 A3/A4 一致，不再明确硬性规定放大倍率（如 2x 或 3x）**。同时拒绝“Canvas 物理尺寸必须等于巨大 ViewBox 尺寸”的强制要求。
* **决策理由**: 这是在“动态清晰度”与“设备上限”之间的极致平衡。由于用户的设备从手机、平板到 5K 显示器跨度极大，如果代码死板地写死 2x 或 3x 放大，在某些超大高分屏上乘以 3 倍依然会诱发内存溢出（OOM）；而在小屏幕上乘以 2 倍导出的图又可能不够清晰。因此，最聪明的策略是：代码只管按比例锁死 A3/A4 画布的宽高比，具体生成多大的像素矩阵，应当交由前端代码在运行时根据 `window.devicePixelRatio` 以及当前 DOM 容器宽度来动态伸缩，从而给出当前环境下的最优解。
* **后续修改文档**: 在 `png-export` 规范中，移除关于 `2x` 或 `3x` 的数值建议。明确导出算法只负责保持 A3/A4 纸张比例，其物理分辨率应当基于当前设备的 DPR（Device Pixel Ratio）作动态柔性伸缩，确保内存安全与导出清晰度的最佳平衡。
