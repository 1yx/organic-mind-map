# Organic Mind Map Phase 2 PRD

## Overview

Phase 2 turns high-quality AI-generated organic mind map images into editable, structured assets.

The user provides:

- a source structure: title, center, branches, children
- a visual reference image generated by GPT Image 2 or similar

The system outputs:

- branch masks
- OCR text boxes and recognized text
- semantic text layers
- doodle masks and crops
- Figma-like visual groups
- editable branch curves
- debug previews and JSON metadata

## Problem

AI image models can produce visually strong organic mind maps, but their outputs are flat raster images:

- branches are not editable
- text is not structured
- doodles and text are fused together
- exact masks are unavailable
- repeated prompt engineering can degrade organic quality into flowchart-like diagrams

Users need a workflow that preserves the quality of the generated visual reference while recovering enough structure to edit and rebuild it.

## Product Principle

Do not force the image model to produce engineering-perfect output.

Use the image model for:

- holistic composition
- organic layout rhythm
- doodle style
- visual inspiration

Use deterministic tooling for:

- layer extraction
- text recognition
- source-structure alignment
- grouping
- editable vector branch curves
- export and correction

## Phase 2 User Flow

```text
1. User prepares source structure
2. User generates or selects best visual reference image
3. User runs local CV extraction
4. System extracts branches/text/doodles
5. System runs OCR and classifies text using source structure
6. System groups doodles with internal doodle text
7. System extracts editable branch curves
8. User reviews debug previews
9. User corrects branch paths and groups
10. System exports structured editable artifacts
```

## Source Structure

The source structure is the semantic truth. Example:

```json
{
  "title": "How Anthropic's Product Team Moves",
  "center": "Anthropic 产品之道",
  "branches": [
    {
      "concept": "极速交付",
      "children": ["研究预览", "跨职能"]
    }
  ]
}
```

OCR is used to locate and read visual text, but source structure decides whether recognized text is:

- title text
- center text
- branch text
- child text
- doodle text
- unassigned text

## Requirements

### R1: Reference Image Input

The system shall accept a raster reference image generated by an AI image model.

The system should preserve the original image for visual comparison and debugging.

### R2: Branch Layer Extraction

The system shall extract colored organic branches into a branch mask.

The system should remove center-card and doodle-color leakage when possible.

The system should output:

```text
branches_mask.png
branches_rgba.png
debug_overlay.png
```

### R3: OCR Text Extraction

The system shall support PaddleOCR for text detection and recognition.

The system shall store OCR boxes with:

```text
id
bbox
text
score
```

The system should support a no-OCR fallback, but PaddleOCR is the preferred path.

### R4: Semantic Text Classification

The system shall classify OCR text against source structure before using spatial heuristics.

The system shall output separate masks for:

```text
title_text
center_text
branch_text
child_text
doodle_text
unassigned_text
map_text
```

### R5: Doodle Extraction

The system shall extract remaining non-branch, non-map-text visual material as doodle components.

The system should output transparent doodle crops and a doodle mask.

The system should repair common OCR-subtraction holes through local post-processing.

### R6: Visual Groups

The system shall group doodle components and doodle-internal text into Figma-like groups.

A group should include:

```text
group id
group bbox
doodle members
doodle_text members
export references
```

Mask separation and spatial binding are both required.

### R7: Editable Branch Curves

The system shall convert branch strokes into editable vector paths.

At minimum, each branch curve should include:

```text
SVG path
stroke color
stroke width
bbox
source points or control points
```

The output should be editable with a pen-tool-like interface.

### R8: Paper.js Prototype

The Phase 2 editable branch prototype shall use Paper.js.

It should support:

- source image visibility toggle
- outline reference visibility toggle
- editable centerline paths
- select/edit segments and handles
- pen mode for extending or creating paths
- opacity control
- reset
- SVG export

### R9: Debuggability

Every extraction step shall produce inspectable artifacts.

Required debug artifacts include:

```text
contact sheet
mask previews
group preview
branch skeleton debug
branch overlay debug
coverage diff where useful
```

### R10: Local-First Operation

The core Phase 2 workflow shall run locally.

Hosted services may be used for experiments or optional refinement, but not for the baseline path.

## Non-Goals

- Do not turn Organic Mind Map into a generic whiteboard.
- Do not depend on AI image model masks as final truth.
- Do not require hosted SAM2 for baseline doodle extraction.
- Do not require exact automatic semantic reconstruction from image alone.
- Do not build a multi-user cloud product in Phase 2.
- Do not make `.omm` visual editing the immediate target unless this pipeline is stable.

## Acceptance Criteria

For the current Anthropic reference image:

- six major branch components are extracted
- source-structure map text is classified into title/center/branch/child categories
- doodle text is separated from map text when not in the source structure
- visual groups are generated with doodle/text membership
- editable branch curves are exported to SVG and Paper.js canvas
- debug previews make errors obvious

## Open Questions

- Should center visual and center text become a dedicated group type?
- Should decorative annotation text have its own category separate from doodle text?
- What is the minimum editable branch quality required before integrating into `@omm/web`?
- Should branch width be represented as constant stroke width, sampled width profile, or editable outline?
- Should promptable SAM2 be evaluated through fal.ai or local SAM2 after local post-processing is mature?
