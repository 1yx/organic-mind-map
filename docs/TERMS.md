# Organic Mind Map Terms

## Canonical Branch Data

`canonical branch data` is the product's standard source-of-truth representation for editable branch geometry.

It is not a debug artifact, not a mask, not a temporary SVG export, and not a Paper.js internal object. It is the structured branch data that the editor saves, loads, renders, edits, exports, and later uses as Phase 3 training/correction data.

## Content Outline

`content outline` is the semantic and visual-intent structure provided by the user (via `content-outline-text`) or generated by an AI agent (via LLM) before visual reconstruction.

It describes what the mind map should say and what each concept should visually suggest. It is not final layout geometry, but it does carry doodle prompts / visual hints so GPT-Image-2 does not invent ambiguous or misleading drawings from short multi-meaning keywords.

In Phase 2, the preferred format for manual input is `content-outline-text`: an indentation-based plain text hierarchy without explicit field names.

Example:

```text
Anthropic 产品之道
  极速交付
    研究预览
    跨职能
```

Interpretation:
- Line 1 (indent 0): Center Visual/Text
- Line 2+ (indent 1+): Main Branches and their recursive subbranches

The content outline is used to:
- guide GPT-Image-2 generation
- classify OCR text
- align extracted text with known concepts
- disambiguate doodle generation for short or multi-meaning concepts
- build canonical `branch` and `subbranch` objects
- detect doodle-local text and `unassigned_text`

Each concept may include a `doodlePrompt` or equivalent visual hint in the canonical JSON form. `content-outline-text` stays field-less for user input, but the normalized `content_outline.json` should preserve richer visual intent.

## content-outline-text

`Content Outline Text` (`content-outline-text`) is the constrained, field-less, indentation-based plain text outline format used by Phase 2 for manual source outline editing.

It is intentionally not YAML. It does not use `-` list prefixes, key/value syntax, or general YAML parsing.

- 2 spaces = one indentation level
- first non-empty indent-0 line = center concept
- indent level 1 = top-level `branch` / BOI
- indent level 2+ = recursive `subbranch`
- blank lines and full-line comments are ignored
- tabs, skipped indentation levels, inline comments, and field syntax such as `center: ...` are not supported

`content-outline-text` is source input. It is parsed into `content_outline.json`, where IDs and visual hints such as `doodlePrompt` are added.

Example canonical concept:

```json
{
  "id": "subbranch_001_001",
  "class": "subbranch",
  "concept": "研究预览",
  "doodlePrompt": "magnifying glass inspecting a preview window with a small lab flask, no text",
  "children": []
}
```

## Branch Outline

`branch outline` is the closed visual boundary of a rendered branch.

It is generated from:

```text
centerline + widthProfile
```

Rule:
```text
canonical branch data stores centerline + widthProfile
renderer generates branch outline
```

## Blank Branch

`blank branch` is an organic branch or subbranch with no keyword text yet.

It is allowed by Buzan's "blank line" rule as a prompt for continued association. Therefore the OMM schema must not require every branch/subbranch to have a non-empty text label.

A blank branch may still have:
- stable ID
- parent relationship
- centerline
- width profile
- color inheritance
- optional doodle prompt or placeholder intent
- later user-filled concept text

## Cloud Boundary

`cloud boundary` is a cloud-like visual enclosure around a complete branch chunk.

It is not a generic rectangle/card. It is a Buzan grouping aid used to strengthen memory chunks around a branch system or subbranch group.

The OMM schema should represent cloud boundaries as first-class visual/grouping objects with:
- stable ID
- referenced branch or subbranch root
- member object references
- outline geometry or generated boundary parameters
- style such as stroke color, fill, opacity, and hand-drawn roughness

## Association Line

`association line` is a non-hierarchical visual connection between two existing OMM objects.

It represents Buzan-style nonlinear association: causality, contrast, complement, dependency, analogy, or another cross-branch relationship that does not belong in the parent/child branch tree.

An association line is independent from organic branches. It should not inherit branch taper rules and should not be treated as a subbranch.

The OMM schema should represent association lines as first-class objects with:
- stable ID
- source object ID
- target object ID
- direction: `none`, `forward`, `backward`, or `bidirectional`
- optional relationship label/type
- curve geometry
- independent style such as thin stroke, dashed/dotted pattern, arrowheads, and color

The editor should provide an independent association-line tool, separate from the organic branch pen tool.

## .omm

`.omm` is the Organic Mind Map document file format. Like `.excalidraw`, it is a product-specific file extension backed by JSON.

There is one OMM file format. Names such as `prediction_omm`, `correction_omm`, and user-facing `.omm` describe where an OMM document came from or what stage it represents; they are not separate file formats.

An OMM document may store:
- content outline
- canonical branch data
- text objects
- doodle groups
- association lines
- cloud boundaries
- embedded or referenced assets needed to reopen the document
- optional masks, debug references, correction records, or provenance depending on the producer and audience

User-facing `.omm` files saved from the editor should not include masks by default. Masks are internal extraction, correction, debug, and training data, not normal user-facing canvas objects.

PNG and SVG exports should be rendered from an OMM document.

## Artifact

`artifact` is a storage-layer file or blob reference managed by the backend.

Artifacts are job outputs, uploads, exports, debug files, or persisted document files. They exist outside or around the OMM document as stored resources with ownership, MIME type, byte size, lifecycle, and access control.

Examples:

- `reference_image`
- `content_outline`
- `prediction_omm`
- `user_omm`
- `correction_omm`
- `mask`
- `debug_overlay`
- `png_export`
- `svg_export`
- `debug_bundle`

API endpoints should use `artifact` when referring to stored files, downloadable blobs, generated outputs, or server-managed references.

An artifact may contain an OMM document, a PNG image, an SVG export, a mask, or a debug bundle. It is not itself a canvas object.

## Asset

`asset` is a business-layer OMM canvas object representing visual material inside a mind map.

Assets are part of the `.omm` document model. They represent doodles, symbols, icons, illustrations, imported visual material, or the center visual. They have OMM object IDs, geometry, role, source references, grouping relationships, masks, and renderer behavior.

Examples:

- center visual asset
- doodle illustration asset
- imported image asset
- icon/symbol asset
- extracted visual component attached to a branch or subbranch

Schema and renderer code should use `asset` when referring to visual objects inside the editable mind map.

Do not use `artifact` for OMM canvas objects. Do not use `asset` for storage-layer files. The boundary is:

```text
artifact = stored file/blob/reference
asset    = OMM document object on the canvas
```

## prediction_omm

`prediction_omm` is an OMM document instance produced by the CV/OCR extraction pipeline.

It contains model and algorithm outputs such as:
- OCR boxes and recognized text
- branch masks and branch centerline predictions
- doodle masks and crops
- initial visual groups
- confidence scores
- extractor metadata and debug artifact references

`prediction_omm` uses the same OMM JSON-backed document format, but its producer is the machine extraction pipeline. It is usually retained by the server for debugging, replay, quality comparison, and Phase 3 training data rather than exported as the user's final polished map.

## correction_omm

`correction_omm` is an OMM document instance produced by an internal/admin correction workflow over a `prediction_omm`.

It contains operations such as:
- confirm
- relabel
- merge / split
- paint / erase
- attach / detach
- move
- reshape_centerline

`correction_omm` uses the same OMM JSON-backed document format, but its producer is the internal correction workflow. It is used for inspect-and-correct workflows, debugging, benchmark evaluation, and Phase 3 training data.

The internal correction UI can derive corrected training truth from:

```text
prediction_omm + correction_omm
```

User-facing OMM exports should generally contain the current editable map state and avoid full internal debug data by default. Server-side systems may still retain `prediction_omm` and `correction_omm` instances for internal quality and Phase 3 training.

## Branch Schema

In Organic Mind Map, a `branch` is the top-level same-color branch system that grows from the center.

In Buzan terms, `branch` is equivalent to a BOI (Basic Ordering Idea): a primary organizing concept that divides the map into a major category. Because `branch` already carries the BOI role, OMM does not need a separate `boi: true` flag for normal top-level branches.

A `subbranch` is any descendant branch that grows from a branch or another subbranch.

Branch identity and display order are separate:

- `id` is stable and constant. For top-level branches, the visible BOI marker can be derived from the stable branch ID.
- `displayOrder` controls layout/read order. It is not necessarily the same as the ID-derived BOI marker.

OMM should not store a separate `orderLabel` field for normal branch numbering.

When an OMM document is created or rebuilt from `content-outline-text`, top-level branch IDs may be assigned from the source list order, e.g. first branch -> `branch_001`. In that source-rebuild context, changing the `content-outline-text` list order changes the ID-derived BOI marker on the next rebuild.

After an OMM document exists, Phase 2 does not support live source-outline reflow into the canvas. Reordering `content-outline-text` in the left-bottom source panel should not immediately mutate existing branch IDs, `displayOrder`, or canvas layout. Live source-outline-to-canvas reflow is a Phase 3 capability.

Within an existing OMM document, reordering the visual/read order should update `displayOrder` without forcing IDs to change.

Top-level branch example:
```json
{
  "id": "branch_001",
  "class": "branch",
  "concept": "极速交付",
  "displayOrder": 10,
  "color": "#fc8b05",
  "thicknessScale": 1.0,
  "mainStroke": {
    "centerline": "M 620 500 C 540 420 460 310 330 250",
    "widthProfile": {
      "type": "root-to-tip-gradient",
      "rootWidth": 38,
      "tipWidth": 12
    }
  },
  "children": ["subbranch_001_001", "subbranch_001_002"]
}
```

## Width Profile

`widthProfile` defines how the visible branch width is derived from the editable centerline.
Phase 2 uses gradient/tapered width:
```json
{
  "type": "root-to-tip-gradient",
  "rootWidth": 38,
  "tipWidth": 12
}
```

## Centerline

`centerline` is the editable path running through the middle of the visible branch.
Centerline direction is canonical:
```text
M point = root side (near center/parent)
path end point = tip side (outer edge)
```
Current preferred representation is an SVG path string.
