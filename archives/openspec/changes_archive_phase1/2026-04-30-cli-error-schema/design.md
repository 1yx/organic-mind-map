# cli-error-schema Design

## Context

The current preview command prints human-oriented errors to stderr and a ready marker to stdout. That works for humans, but it is fragile for Agent CLI workflows that need to parse validation failures and decide whether to regenerate `OrganicTree`, fix JSON syntax, change CLI arguments, or retry server startup.

The desired model is similar to MCP tool responses:

```text
content
  = human-readable summary

structuredContent
  = stable machine-readable result
```

Expected CLI failures should be treated as structured command results, not protocol crashes.

## Goals / Non-Goals

**Goals:**

* Add `omm preview --json` for machine-readable Agent workflows.
* Emit one single-line JSON result envelope to stdout for expected JSON-mode outcomes.
* Represent blocking errors and soft warnings with one `findings[]` shape.
* Preserve default human-readable CLI behavior.
* Preserve exit code semantics for shell compatibility.
* Include preview readiness (`pid`, `url`) in JSON-mode success output.

**Non-Goals:**

* No line/column JSON parser diagnostics in MVP.
* No semantic rewrite suggestions generated by CLI.
* No replacement of human mode.
* No change to browser-side render diagnostics.
* No requirement to add many warning rules in this change.

## Decisions

### JSON mode outputs result envelope, not only errors

`--json` emits a result envelope for success, warning-only success, and expected blocking errors. This avoids splitting success readiness and warning/error metadata across multiple terminal formats.

### Expected JSON-mode results go to stdout

In `--json` mode, expected outcomes are single-line JSON on stdout. That includes `ok: false` results such as contract, quality, capacity, usage, file-read, and server-startup failures. Stderr is reserved for uncaught exceptions or program-level crashes.

### Human mode remains default

Without `--json`, the CLI remains human-friendly:

* success prints `[OMM_SERVER_READY] PID:<pid> <URL>` to stdout,
* errors print readable text to stderr,
* exit codes remain unchanged.

### Findings use severity

Each finding includes:

```ts
severity: "error" | "warning"
```

Warnings do not block preview and produce `ok: true` with exit code `0`. Errors block the relevant command phase and produce `ok: false` with a non-zero exit code.

### Agent actions are explicit

The result envelope includes `agentAction` as a small enum:

```ts
"open-preview"
| "fix-command"
| "fix-json-syntax"
| "regenerate-organic-tree"
| "retry-later-or-change-port"
| "none"
```

This gives Agent CLIs a stable next step without relying on prose.

### Paths use JSON Pointer

Machine findings use JSON Pointer paths such as `/center/concept` and `/branches/0/concept`. Existing internal path formats can be converted at the CLI boundary.

### Repair instructions are constraint-oriented

CLI repair guidance states what constraint failed and what strategy to use. It does not provide concrete semantic replacements or truncate user meaning.

## Result Shape

```ts
type CliJsonResult = {
  schema: "omm.cli.result";
  version: 1;
  command: "preview";
  ok: boolean;
  exitCode: 0 | 1 | 2 | 3;
  agentAction: AgentAction;
  content: Array<{ type: "text"; text: string }>;
  structuredContent: {
    kind:
      | "success"
      | "usage"
      | "file-read"
      | "json-parse"
      | "contract"
      | "quality"
      | "capacity"
      | "server-startup";
    ready?: { pid: number; url: string };
    findings: CliFinding[];
  };
};
```

## Risks / Trade-offs

* **Risk: Agent reads stderr instead of stdout in JSON mode.** -> Document stdout as the only expected result channel for `--json`.
* **Risk: Pretty JSON is easier for humans but harder for agents.** -> Emit single-line JSON in machine mode.
* **Risk: Warnings expand validation scope.** -> Define warning support now, but keep new warning rules minimal and optional for this change.
* **Risk: Path conversion introduces bugs.** -> Add tests for representative nested paths.
* **Risk: Server blocks after success JSON.** -> Print JSON after listener readiness, then keep the process attached as before.
