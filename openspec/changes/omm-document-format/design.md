# omm-document-format Design

## Goal

Define the minimal `.omm` document model needed for Phase 1 read-only rendering while preserving the long-term product rule: one `.omm` equals one paper mind map.

## Document Boundary

`.omm` is an open, local, portable document. It stores semantic content facts plus the browser-computed layout snapshot required to reproduce the exported paper. It must not store Plus cloud service state.

The file should be JSON for MVP simplicity:

```ts
interface OmmDocument {
  id: string
  version: 1
  title: string
  paper: PaperSpec
  organicSeed: string
  rootMap: MindMap
  layout: LayoutSnapshot
  assets: AssetManifest
  references?: ExternalFileRef[]
  meta: DocumentMeta
}
```

## Paper Spec

Only horizontal A3/A4 are valid:

```ts
type PaperKind = "a3-landscape" | "a4-landscape"

interface PaperSpec {
  kind: PaperKind
  widthMm: number
  heightMm: number
}
```

Canonical dimensions:

* A3 landscape: 420 x 297 mm
* A4 landscape: 297 x 210 mm

## Mind Map Model

```ts
interface MindMap {
  id: string
  title: string
  center: CenterVisual
  children: MindNode[]
  relationLinks?: RelationLink[]
  clouds?: CloudOutline[]
  theme?: ThemeState
}
```

For MVP, relation links and clouds may be empty but should be represented so later rendering can add them without changing the file boundary.

The persisted file uses a one-way nested tree. It does not store a flat `nodes` dictionary, `parentId`, or `childIds`. Those indexes are runtime conveniences and should be derived when the application loads the document.

## Node Model

```ts
interface MindNode {
  id: NodeId
  concept: string
  language?: "zh" | "en" | "mixed" | "unknown"
  visualTokens?: VisualToken[]
  imageRef?: AssetRef
  branchStyle?: BranchStyleRef
  children?: MindNode[]
  boi?: BoiTag
  externalRefs?: ExternalFileRef[]
}
```

`concept` is the renderable concept unit. Sibling order is the order of the `children` array. The file does not persist `displayText`; truncation, ellipsis, fading, or other text display adaptations are runtime ViewModel concerns.

## Layout Snapshot

The browser owns final text measurement and layout. Therefore the `.omm` exported by the browser must persist the computed paper geometry so the file represents a stable finished sheet, not only a logical outline.

```ts
interface LayoutSnapshot {
  engineVersion: string
  measuredAt: string
  viewport: {
    widthPx: number
    heightPx: number
    viewBox: string
  }
  environment?: {
    fontFamily?: string
    fontVersion?: string
    userAgent?: string
  }
  center: CenterLayout
  nodes: Record<NodeId, NodeLayout>
  branches: Record<NodeId, BranchLayout>
}

interface CenterLayout {
  box: LayoutBox
}

interface NodeLayout {
  nodeId: NodeId
  textAnchor: Point
  textPathId?: string
  textBox: LayoutBox
  visualBox?: LayoutBox
}

interface BranchLayout {
  nodeId: NodeId
  branchPath: string
  textPath: string
  strokeWidthStart: number
  strokeWidthEnd: number
}

interface LayoutBox {
  x: number
  y: number
  width: number
  height: number
}

interface Point {
  x: number
  y: number
}
```

The layout snapshot references semantic nodes by stable `nodeId`. It may be regenerated by a future layout pass, but Phase 1 export must write it so `.omm` can reproduce the browser-computed paper and later support Figma/SVG/PDF export without rerunning layout from scratch.

## Center Visual

```ts
interface CenterVisual {
  mode: "image" | "styled-text" | "hybrid"
  titleText: string
  imageRef?: AssetRef
  visualHint?: string
  minColorCount?: number
  complianceState: "draft" | "needs-visuals" | "compliant"
}
```

MVP generates a compliant center through a built-in template or styled built-in composition. User-uploaded center or node images are not part of Phase 1.

Long-term strict mode can require at least three colors for compliant center visuals. Phase 1 allows a single-color controlled SVG center visual when it is still a real image or visual symbol rather than plain text.

## Asset Handling

Phase 1 `.omm` avoids Base64 image bloat by supporting built-in assets only. User-uploaded center images and node images are deferred beyond MVP.

Built-in assets:

* store a stable built-in asset ID only
* actual SVG/image/template data lives in the Web renderer static asset registry
* keeps `.omm` lightweight and text-friendly

```ts
interface ImageAsset {
  id: string
  mimeType: string
  source: "builtin"
  builtinId: string
}
```

For `source: "builtin"`, `builtinId` is required and embedded payload data must be omitted. External paths can be recorded as optional references, but they are not rendering dependencies in Phase 1.

## Validation

Validation should ensure:

* document version is supported
* paper kind is valid
* center visual exists
* center is not plain text compliant unless styled visual requirements are met
* nested node ids are unique across the full tree
* sibling ordering is preserved by `children` arrays
* no persisted `parentId`, `childIds`, or flat node dictionary is required
* layout snapshot exists after browser export
* layout snapshot references only existing node ids
* layout paths and boxes are present for renderable nodes and branches
* built-in asset references point to known registry IDs
* uploaded/Base64 assets are not present in Phase 1 `.omm`

## Non-stored State

Do not store:

* editor selection
* drag sessions
* undo stacks
* `displayText` strings
* text truncation or ellipsis strings
* cloud permissions
* RAG indexes
* source file snapshots
* source object mappings
* submap navigation state

## Risks

* Making the schema too large before rendering proves itself.
* Storing external paths that break portability.
* Reintroducing Base64 uploads before there is a compression/package strategy.
* Confusing runtime store indexes with static document shape.
* Confusing browser display artifacts with semantic document content.
* Confusing `concept` with raw source text.

## Decisions

* JSON is sufficient for MVP.
* Keep versioning explicit.
* Store semantic content facts plus the browser-computed layout snapshot required to reproduce the exported paper.
* Persist semantic nodes as a one-way nested tree; derive flat indexes and parent references at runtime.
* Do not persist `displayText`; keep truncation and display adaptations in runtime ViewModels.
* Built-in templates are referenced by ID.
* User-uploaded image assets and Base64 embedding are deferred beyond Phase 1.
