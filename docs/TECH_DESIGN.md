# Technical Design: Organic Mind Map

## 1. 设计目标

本技术设计服务于以下已确定产品约束：

* MVP 先实现 Agent CLI 调用 agent skill、项目 CLI 校验并启动本地 Web、浏览器端布局、`.omm`/PNG 导出，不实现 `.omm` 可视化编辑。
* 严格博赞模式是默认且唯一主产品形态。
* 一个 `.omm` 文件只对应一张思维导图，即一张纸。
* 纸张规格固定为 A3/A4 横向，不支持无限画布作为标准有机导图模式。
* 中心图像是必需对象，不能退化为纯文本中心。
* 一个概念单元一分支通过本地启发式与非打断式 AI 建议实现，而不是机械词数限制或弹窗阻断。
* AI 收敛不得静默改变用户语义；高风险输入需要保护原文或确认。
* 用户手动拖拽是布局意图输入；如果破坏博赞结构，系统自动纠偏。
* Mermaid / Excalidraw / Figma 导入后必须通过净化管线重排为博赞结构。
* A3/A4 打印与导出一致性是一等公民能力，必须前置设计。
* 渲染需要受控随机性，同一张图稳定复现，同一模板下不同图具备有机差异。
* 曲线文字呈现与水平文字编辑分层处理。
* 移动端和平板压感笔是 Phase 5+ 方向，底层接口需要预留。
* 当前不做产品内子图系统、面包屑回溯、颜色继承、引用保护、断链检测或自动修复。
* 不做实时协作，所有架构以单人编辑可靠性为中心。

这意味着技术设计的目标不是先做完整编辑器，而是先打通“长文本 -> Agent CLI 调用 skill -> 简短层级列表 -> 项目 CLI 校验与启动 Web -> 浏览器布局 -> `.omm`/PNG 导出”的稳定链路。完整可视化编辑器是后续阶段。

### 1.1 MVP 技术范围

MVP 包含：

* **agent skill 输入协议**: 由 Gemini CLI / Codex CLI / Claude Code 等 Agent CLI 调用，将长文本切成符合“一个概念单元一分支”的简短层级列表。
* **CLI 校验与启动器**: 接收简短层级列表，执行契约校验和容量阈值检查；超限时返回可供 Agent 重试的错误，合格时启动本地 Web。
* **本地 Web 服务**: CLI 启动本地服务，将合格 payload 提供给浏览器。
* **浏览器端布局与只读 SVG 渲染**: 浏览器基于 DOM/SVG 真实文本测量生成布局，渲染 A3/A4 横向纸张、中心图像、多色主分支、锥形曲线、路径文字和基础素材。
* **本地导出**: 从本地预览导出 `.omm` 与 PNG 图片。

MVP 不包含：

* `.omm` 可视化编辑
* 拖拽、重组、撤销重做
* 路径文字编辑浮层
* 用户自定义演示路径
* 云同步、账户、Plus 服务

## 2. 总体架构

系统分为 7 个核心层次：

1. **文档模型层**
   单文件单图的数据结构，定义节点、分支、中心图像、关系链接、云朵边界等对象。

2. **编辑状态层**
   管理选中态、输入态、AI 收敛中状态、拖拽态、撤销重做。

3. **规则与收敛层**
   管理严格博赞规则，包括一个概念单元一分支、中心图像、留白约束、自动纠偏和 AI 语义保护。

4. **布局求解层**
   依据节点顺序、概念单元长度、图像尺寸、纸张边界和留白约束生成几何结果。

5. **渲染与测量层**
   使用 SVG 作为主渲染介质，同时维护稳定的文字测量、路径测量和打印映射。

6. **导入导出层**
   支持 Mermaid / Excalidraw / Figma / JSON / PNG / SVG / PDF 的读写与转换。

7. **容器与导航层**
   `.omm` 文件之上的“册子 / 文件夹”组织层，以及简单外部链接跳转。

## 3. 技术选型

### 3.1 前端基础

* Vue 3
* TypeScript
* Pinia
* Vite

这些选型保持不变。MVP 的 Web 端是只读渲染预览，不需要完整编辑器状态；后续可视化编辑阶段再引入完整的编辑状态与命令模型。

### 3.2 渲染层：主 SVG

选择纯 SVG 作为主渲染层，原因如下：

* 直接支持 A3/A4 矢量输出。
* 支持路径文字与分支几何共存。
* 可与 Vue 组件和响应式状态较自然结合。
* 便于导出 SVG，并作为 PDF 导出的中间表示。

但实现上不能把“SVG 好操作”理解成“所有交互都直接绑 DOM 即可”。系统必须区分：

* **显示层**：最终可见几何
* **命中层**：用于选择、拖拽、hover 的辅助几何
* **编辑层**：用于文本输入、临时编辑框、加载态
* **测量层**：用于稳定文本与图像占位计算

### 3.3 明确不做的事情

* 不做实时协作
* 不做多人 presence
* 不做协同补丁同步
* 不做复杂外部引用同步
* 不做开放自由布局模式作为核心产品形态
* 不做产品内子图下钻、面包屑、引用保护或断链自动修复
* 不做移动端首发体验

## 4. 文件模型与容器模型

### 4.1 `.omm` 文件语义

一个 `.omm` 文件只表示一张图。它不承载多张 map 集合。

建议的顶层结构：

```ts
interface OmmDocument {
  id: string
  version: number
  title: string
  paper: PaperSpec
  organicSeed: string
  rootMap: MindMap
  assets: AssetManifest
  references: ReferenceLink[]
  meta: DocumentMeta
}
```

`paper` 只允许 A3/A4 横向规格。`organicSeed` 是文档级稳定随机源，用于驱动分支微抖动、曲率偏移、锥形比例差异和手绘感模拟；同一文件重复打开必须渲染一致。

### 4.2 图模型

```ts
interface MindMap {
  id: string
  title: string
  center: CenterVisual
  rootChildren: NodeId[]
  nodes: Record<NodeId, MindNode>
  relationLinks: RelationLink[]
  clouds: CloudOutline[]
  presentationSequence?: PresentationStep[]
  theme: ThemeState
}
```

### 4.3 节点模型

`children` 必须是有序数组，而不是集合。

```ts
interface MindNode {
  id: string
  parentId: NodeId | null
  branchSide: "left" | "right" | "radial"
  order: number
  rawInput: string
  keywordText: string
  displayText: string
  language: "zh" | "en" | "mixed" | "unknown"
  keywordState: KeywordState
  overflowState: TextOverflowState
  visualTokens: VisualToken[]
  imageRef?: AssetRef
  branchStyle: BranchStyleRef
  childIds: NodeId[]
  boi?: BoiTag
  externalRefs: ExternalFileRef[]
  layoutHint?: NodeLayoutIntent
}
```

说明：

* `rawInput` 保存用户原始输入。
* `keywordText` 保存最终用于渲染的概念单元。
* `displayText` 保存当前渲染态文本，可能因为纸张空间限制而被可见裁切。
* `keywordState` 表示是否正在 AI 收敛中。
* `overflowState` 表示文本是否超出当前分支可见长度，用于渲染态裁切，不主动提示用户。
* `childIds` 是稳定有序数组，表示树结构顺序；演示顺序由独立的 `presentationSequence` 决定。

```ts
type TextOverflowState =
  | "fits"
  | "render-clipped"
```

```ts
interface PresentationStep {
  targetNodeId: NodeId
  mode: "node" | "subtree"
}
```

### 4.4 中心图像模型

中心不能只是普通字符串，必须是可图像化对象。纯普通文本中心只能作为草稿态或待补图状态，不能作为严格模式合格导图。

```ts
interface CenterVisual {
  mode: "image" | "styled-text" | "hybrid"
  titleText: string
  imageRef?: AssetRef
  stylePreset?: CenterStylePreset
  minColorCount: number
  composition: CenterComposition
  complianceState: "draft" | "needs-visuals" | "compliant"
}
```

支持 3 种中心表现：

* 真图片
* 多色艺术化文字或视觉构图
* 图文混合中心构图

合格中心图像必须至少 3 色，并具备明确图像化轮廓或视觉符号组合。免费版应通过基础中心模板、图标拼贴、多色视觉构图、本地上传图片或图文混合完成，不依赖 Plus AI。

### 4.5 资源模型

```ts
interface AssetManifest {
  images: Record<string, ImageAsset>
  symbols: Record<string, SymbolAsset>
  templates: Record<string, TemplateAsset>
}
```

`.omm` 必须保证独立打开后图片完整。必要图片资源应内嵌或随 `.omm` 自包含保存；外部路径只能作为来源记录或可选链接，不能作为打开导图的唯一依赖。

图像资源需要区分来源：

* 内置基础模板
* 预生成免费模板
* 免费本地上传资源
* Plus AI 生成资源
* Plus 云端管理或高级处理资源

### 4.6 外部链接

当前不做产品内子图下钻、面包屑回溯、颜色继承或引用保护。只保留简单外部链接 / 路径引用能力。

```ts
interface ExternalFileRef {
  kind: "reference"
  targetPath: string
  label?: string
}
```

文件被用户在系统外移动或重命名后，链接断裂是可接受的，由用户自行处理。不引入项目级清单、全局 ID 扫描、断链检测或自动修复作为 MVP 要求。

### 4.7 容器模型

容器不进入 `.omm` 文件内部定义，而存在于文件系统或 Web 工作区中：

* Web：`Book / Workspace`
* 本地：文件夹
* 插件场景：文件夹

技术上应保持：

* 文档模型独立
* 容器层只负责组织、浏览、跳转

## 5. 编辑状态与命令模型

本章节属于可视化编辑阶段，不进入 MVP。MVP 只需要从 `.omm` 文档生成只读渲染状态和导出状态。

### 5.1 单人编辑状态

```ts
interface EditorState {
  selection: SelectionState
  inputSession?: InputSession
  dragSession?: DragSession
  layoutPreview?: LayoutPreview
  exportState?: ExportState
  undoStack: Command[]
  redoStack: Command[]
}
```

### 5.2 为什么仍然要命令模型

虽然不做协作，但仍然必须有命令模型，因为以下操作不是简单字段修改：

* 调整兄弟顺序
* 自动纠偏
* AI 关键词建议与用户确认
* 导入自动重排

建议把“用户操作 + 系统求解结果”记为单个事务：

* 用户输入长句
* AI 给出关键词建议或确认后的收敛结果
* 布局更新

或者：

* 用户拖拽节点
* 布局引擎自动回正

这些都应作为可撤销事务处理。

## 6. 输入与关键词收敛管线

MVP 中，关键词收敛主要发生在 agent skill 阶段：Agent CLI 调用 skill，将长文本切分为简短层级列表，再交给项目 CLI 校验和启动本地 Web。Web 端暂不提供交互式文本输入和收敛 UI。

### 6.1 目标

满足以下同时成立：

* 不阻断输入
* 不弹窗警告
* 最终尽量保持一个概念单元一分支
* 不让长句破坏分支的物理比例和有机形态
* 保留用户原始输入的主权
* 不静默改变用户语义

### 6.2 状态机

```ts
type KeywordState =
  | "clean"
  | "raw-input"
  | "compressing"
  | "compressed"
  | "compress-failed"
```

### 6.3 管线步骤

1. 用户输入任意文本
2. 保存 `rawInput`
3. 本地轻规则先做预裁剪：
   * 去除多余空白
   * 识别语言
   * 判断是否为句子化、解释化、提纲化输入
   * 识别概念单元长度，并测量其理想分支长度与纸张最大可用长度
4. 免费版优先使用本地启发式生成候选 `keywordText`
5. Plus 版若需要语义压缩，进入 `compressing`
6. 异步调用 AI 关键词收敛能力
7. 返回候选 `keywordText`
8. 若输入包含否定、数字、人名、日期、专有名词、因果/条件限定等高风险信息，保护原文或等待用户确认
9. 更新 `displayText`、`overflowState`、布局与渲染

### 6.4 UI 表达

只提供必要交互状态，不做评分式、审查式、警告式提示：

* 节点旁轻量 spinner
* “思考中”或等价视觉状态
* 低风险收敛可自然替换显示词
* 高风险收敛需要保护原文或用户确认
* 溢出时渲染态直接可见裁切，不主动提示
* hover、点击或撤销命令可查看/恢复 `rawInput`

不做：

* modal
* toast 阻断
* 合规检查面板
* 评分式或审查式提示

### 6.5 AI 失败回退

若 AI 收敛失败：

* 保留原始输入
* 用本地启发式做一次降级裁剪
* 标记为 `compress-failed`
* 允许继续编辑，不持续打扰用户

### 6.6 分离式路径文字编辑

最终渲染使用 SVG `textPath`，但编辑时不在曲线上直接处理光标、选区和输入法。

交互模型：

1. 双击路径文字或按 `F2`
2. 在节点附近打开水平输入浮层
3. 输入浮层修改 `rawInput`，后台实时更新路径文字预览
4. 按 Enter 或失焦后提交事务
5. 输入浮层消失，文字平滑贴合分支骨架路径

技术要求：

* 输入浮层属于编辑层，不进入最终导出几何
* 输入法、选区、复制粘贴都交给原生水平文本控件处理
* 路径文字只负责呈现、测量和空间占比反馈

## 7. 布局与自动纠偏

MVP 只需要只读布局求解：根据 `.omm` 的树结构、中心图像、概念单元长度、纸张尺寸和基础视觉规则生成稳定几何结果。用户拖拽意图、自动纠偏和编辑态预览属于后续可视化编辑阶段。

### 7.1 布局目标

布局求解同时满足：

* 中心图像占位清晰
* 主分支向纸张有效空间展开
* 线字等长
* 留白保护
* 兄弟顺序稳定
* 自动纠偏
* 物理纸张边界内可打印
* 内容过密时进入不可容纳状态，而不是无限缩小或引导产品内子图化

### 7.2 布局输入

布局引擎的输入不能只依赖树结构，还必须依赖：

* 纸张规格 A3/A4
* 中心图像包围盒
* 节点顺序
* 关键词测量尺寸
* 图像/符号尺寸
* 手动拖拽意图
* 颜色与主分支侧向
* 文档级 `organicSeed`
* 文本裁切状态与不可容纳状态

### 7.3 建议的求解管线

1. **中心占位阶段**
   计算中心视觉对象的安全包围区域。

2. **主分支分区阶段**
   左右主分支或径向主分支按顺序分配角度区间。

3. **递归子树布局阶段**
   子树按照顺序与长度需求递归展开。

4. **碰撞与留白阶段**
   对文字、图像、分支形状做碰撞检测与局部拉开。

5. **边界纠偏阶段**
   检查是否逼近纸张边界并回收角度或压缩子树。

6. **空间耗尽处理阶段**
   当局部密度无法合法容纳时，进入不可容纳状态。用户需要精炼概念、删除内容、调整结构、另开独立 `.omm` 或手动添加外部链接。

7. **自动回正阶段**
   若用户拖拽破坏规则，则重新求解并回正到合法布局。

### 7.4 手动拖拽语义

手动拖拽不是绝对坐标编辑，而是“布局意图输入”。

例如拖动一个节点时：

* 系统记录新的目标侧向、相对顺序或目标角度偏好
* 不直接永久保留违规位置
* 求解器重新生成合法几何结果

### 7.5 纠偏原则

自动纠偏优先级建议：

1. 保持树结构与顺序
2. 保持中心清晰
3. 保持留白
4. 保持线字等长
5. 尽量尊重拖拽意图

### 7.6 受控随机性

布局结果分为确定性大结构与有机小扰动：

* 大结构由树、顺序、纸张、中心占位和约束求解决定
* 小扰动由 `organicSeed` 派生，影响分支弯曲微调、锥形比例、抖动频率和手绘模拟

随机扰动必须满足：

* 不破坏碰撞检测、留白和节点/分支空间硬约束
* 不影响同一文件的可重复导出
* 在打印和屏幕渲染中使用同一套派生参数

## 8. 文字、图像与分支测量

### 8.1 为什么这一层必须前置

核心规则都依赖测量：

* 线字等长
* 留白控制
* 路径文字可读性
* A3/A4 导出一致性

### 8.2 测量原则

编辑态、预览态、导出态必须尽量共用同一套测量逻辑。

建议建立统一测量服务：

```ts
interface MeasurementService {
  measureText(input: TextMeasureInput): TextMeasureResult
  measurePathText(input: PathTextMeasureInput): PathTextMeasureResult
  measureBranchTextLimit(input: BranchTextLimitInput): BranchTextLimitResult
  measureImage(input: ImageMeasureInput): Box
  measureCenterVisual(input: CenterMeasureInput): Box
}
```

### 8.3 文字测量

必须处理：

* 字体加载完成前后的差异
* 中英文混排
* 字重与字号变化
* 路径文字的有效长度
* 当前分支物理长度对应的可显示字符范围

建议：

* 固定一套默认字体策略
* 首次测量后缓存结果
* 字体未就绪时使用近似值，加载后重算布局

### 8.4 分支几何

锥形分支不应被视为简单 SVG stroke。

建议：

* 分支使用闭合 path 形状渲染
* 单独维护中心骨架路径，供 `textPath` 使用
* 单独维护命中 path，供交互使用

即：

* `branchShapePath`
* `branchSpinePath`
* `branchHitPath`

这三者职责不同。

## 9. 渲染结构

建议渲染顺序：

1. 纸张背景与边界
2. 云朵边界
3. 关系链接
4. 分支形状
5. 分支文字路径
6. 分支图像/符号
7. 中心视觉对象
8. 交互态高亮与编辑层

说明：

* 中心视觉对象应始终保持视觉中心权重。
* 关系链接应避免遮挡关键文字，可在必要时采用局部避让。
* 盒状节点禁止出现，编辑输入应通过临时浮层完成，而不是最终渲染形态。
* 分支形态使用文档级有机种子派生参数，保证有机差异可重复。
* 文本溢出在最终渲染层遵守可见裁切；裁切只影响文本可见范围，不得绕过节点/分支整体空间硬约束。

---

## 10. 导入管线

### 10.1 总原则

导入不是“照搬外部结构”，而是“把外部结构净化并重构为一张新的博赞导图”。系统内不长期保存非博赞几何结构，也不在 `.omm` 中保留源结构、源对象映射或源文件快照。

### 10.2 管线步骤

1. **源解析**
   解析 Mermaid / Excalidraw / Figma 的原始结构。

2. **中间归一化**
   转换为统一的 `ImportedStructure`。

3. **语义识别**
   推断中心主题、树结构、顺序、节点文本。

4. **关键词收敛**
   对长文本做关键词压缩。

5. **中心图像构建**
   为中心主题生成艺术化中心构图工作流。

6. **博赞重排**
   应用颜色、边界、分支方向、留白、线字等长规则。

7. **必要状态**
   对草稿态或待补图态保留必要交互状态，不做合规检查面板。

8. **冲突解决**
   对层级过深、空间不足等内容，提供 AI 建议、手动精炼、调整结构、另开新图或外部链接的处理入口。

### 10.3 重要取舍

导入后结果不追求和源文件视觉完全一致，而追求：

* 结构可用
* 规则合规
* 可继续编辑
* 可打印导出
* 不可逆追溯到源文件结构

---

## 11. 导出管线

### 11.1 导出类型

* `.omm`
* JSON
* SVG
* PNG
* PDF
* Semantic Layered SVG
* Mermaid 二级/多级列表

### 11.2 打印与物理尺寸

必须明确以下映射：

* 纸张毫米尺寸
* SVG `viewBox`
* 屏幕坐标单位
* PNG 输出 DPI
* PDF 物理页面尺寸

建议：

* 内部统一使用物理无关文档坐标
* 在导出时由纸张规格映射到具体输出尺寸

### 11.3 SVG / PDF 一致性

建议：

* SVG 作为主导出中间表示
* PDF 导出尽量基于 SVG 转换而不是重写一套布局

### 11.4 自由导出出口

这是“自由再编辑出口”，不是保持博赞规则的继续编辑链。

要求：

* 导出带语义化图层和内嵌素材的高质量 SVG
* 保留主要几何、颜色、文字、图像
* 尽量转换为 Figma、Photoshop 等外部工具可继续调整的对象
* 不要求在外部工具中继续维持严格博赞约束

## 12. 导航与容器体验

因为当前不做产品内子图系统，导航主要依赖：

* 文件夹列表
* 册子列表
* 文件链接跳转

技术要求：

* 简单外部链接优先采用文件名或相对路径
* 文件移动或重命名导致断链时，由用户自行处理
* 不做断链检测、自动修复、重命名重构或全局 ID 扫描
* 应支持“册子 / 项目”的虚拟容器视图，但不把容器状态写入单个 `.omm` 文件

## 13. 付费能力的技术边界

### 13.1 免费版

* 基础符号库
* 基础中心图模板
* 预生成免费模板图像
* 本地图片上传
* 严格博赞核心编辑能力
* 本地启发式关键词建议
* 固定纸张 SVG / PNG / PDF 导出
* Mermaid 结构导出

### 13.2 Plus

* AI 生图
* 高级素材库
* 高级中心图模板
* 图片风格化、去背、高清处理、云端资产管理
* AI 关键词收敛增强
* 跨 `.omm` 文件的语义检索与 RAG 问答
* Web 交互分享页发布、缩放与密码保护；分享页保持 A3/A4 纸张比例
* Obsidian / Logseq / Notion 双向同步插件属于后期路线，不进入 MVP 或近期核心设计
* 云同步、版本历史、冲突检测与团队共享库

技术上要保证：

* 免费版可以不依赖 Plus 能力完成完整中心图工作流
* Plus 增强视觉质量和数据管理效率，但不改变文档基础格式
* Plus 的语义索引、RAG 缓存、权限、团队成员、发布状态、分享密码、同步版本、冲突记录、审计日志等服务状态存放在云端服务器，必要时可使用旁路 workspace / manifest，不写入 `.omm` 基础格式

## 14. 风险与控制点

### 14.1 最大风险

1. 把布局实现成“树坐标分布”，而不是“约束求解”
2. 把 AI 收敛做成可有可无的小功能，而不是核心输入机制
3. 把中心图像退化为普通文本加背景
4. 把导出做成与编辑态不一致的第二套几何
5. 在当前阶段引入子图下钻、面包屑、引用保护等非必要复杂度
6. 把自由拖拽实现成永久绝对坐标，绕过布局求解
7. 把 Plus 能力做成改变文件格式的封闭数据层

### 14.2 对策

* 先做文档模型与测量层，再做视觉 polish
* 先定义事务型编辑流程，再接 UI
* 先保证 A3/A4 输出一致，再追求更多互动动画
* 先实现原子文件自包含资源，再做云同步和团队库

## 16. 结论

当前产品的核心不是“开放绘制”，而是“严格博赞规则下的高质量单人创作”。因此技术设计的中心也不应是传统脑图编辑器套路，而应是：

* 单文件单图
* 规则驱动输入
* 约束驱动布局
* 测量驱动渲染
* 导出驱动一致性

只要这 5 条站稳，后续 UI、动画、模板和 AI 增强才能建立在可靠底座上。
